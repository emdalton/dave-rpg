## RAG (Retrieval-Augmented Generation)

**What it is.** RAG is a technique for giving a language model access to information that isn't in its context window. Rather than stuffing all potentially relevant content into the prompt (which is costly and may exceed token limits), you store that content externally, then at query time *retrieve* the most relevant pieces and inject them into the prompt alongside the question. The retrieval step is typically semantic — the system converts both stored records and the current query into numerical vectors (embeddings), and finds the stored records whose vectors are most similar to the query vector. Similarity here captures meaning, not just keyword overlap.

The "augmented generation" part is simply that the LLM's generation is augmented by this retrieved context. The LLM doesn't "search" anything itself — it receives the retrieved text as part of its input and reasons over it.

**How it relates to DAVE.** DAVE currently uses a simple recency-based approach: the Pass 2 context packet includes the N most recent `action_log` entries (configurable via `DAVE_PASS1_RECENT_ACTIONS`, default 5; `DAVE_ACTION_LOG_MAX_ROWS` caps the stored log at 50 rows). For short modules, this is fine — relevant history and recent history are usually the same thing.

In a longer-running game, this breaks down. If a player made a promise to an NPC twenty turns ago, or learned a secret that now becomes relevant, the recency window won't surface it. A promise from turn 5 is invisible to Pass 2 by turn 30. The proposed fix (Feature 23 in `docs/future_features.md`) is to embed each `action_log` entry as a vector at write time, and at context assembly time (in `context.py`) retrieve the K entries most semantically similar to the current action — not just the most recent ones. These retrieved entries would populate the existing `recent_actions` field in the Pass 2 packet.

The same technique could be applied to `location_detail` records (generated descriptions accumulated across visits) once a module's location graph grows large enough that including all details in every packet becomes unwieldy.

**What would be required to implement it in DAVE.** An embedding model (could be a small local model, or an API call), a vector index (possibly SQLite-based via an extension like `sqlite-vss` or `sqlite-vec`), and a retrieval step added to `context.py`'s `build_pass2_packet()`. DAVE's current architecture is already well-shaped for this: the `action_log` table has the right structure (text `narrative_beat` per turn, timestamps, character references), and the point of insertion in the pipeline is clear (`recent_actions` in the context packet).

**When it becomes relevant for DAVE.** Not yet needed for Hidden Hostel, Meryton, or the planned Wonderland module at current scale. The trigger would be either: (a) a long campaign where time-ordered recency produces noticeably worse adjudication than a semantic retrieval pilot, or (b) a module's `location_detail` volume grows large enough to require selective retrieval.

## Model Parameters vs. Training Corpus Size

**Parameters** are the learned weights in a neural network — the numbers adjusted during training that encode what the model "knows." Parameter count is roughly a measure of model *capacity*: how complex the representations it can form, how much it can hold. A 7B parameter model has seven billion such weights.

**Training tokens** are the amount of text the model was trained on, measured in tokens (roughly 0.75 words each). This is the model's *experience*. Parameters and training tokens are independent axes: a large model trained on little data has wasted capacity; a small model trained on enormous data eventually hits diminishing returns.

**The Chinchilla result (DeepMind, 2022)** is the main public reference point for the relationship between them. Prior large models — GPT-3 at 175B parameters was the canonical example — were found to be significantly *undertrained*: too much compute budget spent on parameters, too little on training tokens. The compute-optimal ratio Chinchilla identified was roughly **20 training tokens per parameter**. For a 7B parameter model, that suggests an optimal training run of around 140 billion tokens.

In practice, post-Chinchilla thinking has pushed further toward more tokens still. Llama 3 8B (roughly 7B-class) was trained on approximately 15 trillion tokens — an order of magnitude beyond Chinchilla-optimal — on the reasoning that inference is cheap relative to training, so you want the best possible model at a given serving size even if it costs more compute to produce.

**Salamandra 7B and the Common Corpus.** The Common Corpus is approximately 500 billion tokens of permissively licensed and public domain text. At 7B parameters, that's roughly 71 tokens per parameter — above Chinchilla-optimal, so the model is not undertrained relative to its size. It is, however, an order of magnitude smaller than what well-resourced labs now train comparable models on.

Whether this gap matters for DAVE depends on domain coverage, not raw scale. Common Corpus includes Project Gutenberg (the complete Austen, 19th century literary prose broadly), Wikipedia, and a substantial multilingual component — exactly the domains DAVE's modules draw on. A model that has deeply absorbed high-quality literary text may outperform a larger model diluted by web-scrape noise on Pass 2 adjudication (social/psychological reasoning) and especially Pass 3 prose rendering. The planned Salamandra 7B pilot (Feature 19a in `docs/future_features.md`) is the right way to find out: run the Hidden Hostel scenario and compare adjudication quality and prose against the Haiku baseline.

Anthropic does not publish precise parameter counts or training corpus sizes for its models. Most commercial vendors similarly reserve this information. The Chinchilla paper, Llama model cards, and Common Corpus documentation are the primary public sources for the numbers cited here.

## The Inference Stack: From Model File to API

Running a local LLM involves several layers between the model weights on disk and the API call your application makes. Understanding what each layer does helps when deciding what to run on a cloud server.

**Base model / foundation model** — these terms are interchangeable. The raw product of pre-training: the model has seen an enormous corpus and learned to predict the next token. It can complete text fluently but does not follow instructions reliably. Not suitable for DAVE directly.

**Instruct model (also: chat model)** — a base model further trained to follow instructions. This is done via supervised fine-tuning on instruction-response examples, usually followed by a preference alignment step (historically RLHF — Reinforcement Learning from Human Feedback; increasingly DPO — Direct Preference Optimization, which achieves similar results more simply). Alignment is also where safety behaviors are instilled. HuggingFace labels these `-instruct` or `-chat`. For DAVE's three passes, instruct models are required — base models will not reliably produce structured JSON or follow complex prompt instructions.

**Quantization** — the model's weights are floating-point numbers. Quantization reduces their precision (16-bit float → 8-bit integer → 4-bit integer) to reduce memory and compute requirements. A Salamandra 7B model in 16-bit precision requires roughly 14GB of GPU memory; in 4-bit, roughly 4–5GB. Quality degrades with lower precision: 8-bit is nearly indistinguishable from full precision; 4-bit shows some degradation on nuanced reasoning; below that (Q2/Q3), outputs become noticeably flatter and may be incoherent. The analogy to numerical precision in calculations is accurate — small per-weight rounding errors compound across millions of operations.

**Model format** — HuggingFace stores models in `safetensors` format (raw float16/32 weights). Ollama uses `GGUF` format, the native format for `llama.cpp`. GGUF files bake in the quantization and package everything needed into a single file. On HuggingFace, files named `Q4_K_M.gguf` or similar are quantized GGUF files ready for Ollama. The naming encodes the quantization method and variant (`K_M` preserves more accuracy in the most sensitive weights than a plain `Q4`).

**Inference engine** — the runtime that executes the model. Ollama wraps `llama.cpp`, a highly optimized C++ library. For higher-throughput production use, `vLLM` is the more common alternative — it supports continuous batching (serving multiple concurrent requests efficiently) and exposes an OpenAI-compatible API. HuggingFace's `Text Generation Inference` (TGI) is another option in the same tier. For DAVE at single-player scale, Ollama is entirely sufficient.

**Serving layer / API** — Ollama exposes a REST API at `localhost:11434`. Running Ollama on a cloud server and exposing that endpoint makes it a drop-in replacement for the Anthropic API from DAVE's perspective. `vLLM` exposes an OpenAI-compatible API, meaning `engine/llm/ollama.py` could be made nearly identical to `engine/llm/claude.py` in call shape. Designing the Ollama stub to be OpenAI-API-compatible from the start would make it easy to swap inference backends later.

**Chat template** — each instruct model was fine-tuned with a specific prompt format: Llama 3, Mistral Instruct, and Salamandra each use different delimiters and special tokens for system messages, human turns, and assistant turns. Sending raw text without the expected wrapper degrades quality because the model's fine-tuning assumed the format. Ollama handles this automatically for known models via a `TEMPLATE` field in its `Modelfile`; a custom or uncommon model may require explicit template configuration. This is a practical gotcha worth being aware of when setting up Salamandra.

**Ollama's role, summarized.** Ollama is to a local LLM roughly what Apache httpd combined with a process manager (uWSGI, Gunicorn) is to a web application: it's the always-running daemon that receives API requests, manages model loading and memory residency, routes requests to the right model, and returns responses in a standard format. The model file on disk is inert without it.

**For DAVE specifically.** Three LLM calls per turn means per-turn latency is roughly 3× a single call, so inference speed matters more than in a one-call-per-turn system. Pass 1 and Pass 2 require reliable structured JSON output — a known weak point in smaller models; Ollama's `"format": "json"` parameter helps (already on the pending-work list). The Pass 2 context packet can grow large in complex scenes, so the model's effective context window length matters.

## Fine-Tuning

**What fine-tuning is.** Fine-tuning takes an existing trained model and continues training it on a new, smaller, more specific dataset to improve its performance on a particular task or in a particular style. Full fine-tuning (adjusting all parameters) approaches pre-training cost at scale and is rarely practical for 7B models outside well-funded labs. The more relevant concept is **parameter-efficient fine-tuning (PEFT)**, where the original weights are frozen and only a small set of additional parameters are trained on top.

**LoRA (Low-Rank Adaptation)** is the dominant PEFT method. Instead of updating the full weight matrices, LoRA adds small "adapter" matrices that capture the task-specific adjustments. For a 7B model, a LoRA run trains a few million parameters instead of seven billion — feasible on a single consumer GPU (e.g. an RTX 4090 with 24GB VRAM) in hours to a day, depending on dataset size. **QLoRA** combines this with quantization, fine-tuning a 4-bit model, reducing memory requirements further. Inference cost afterward is essentially unchanged from the base model; the adapter adds minimal overhead and loads alongside the base weights.

**Types of fine-tuning by technique.** Your understanding of RLHF is correct. In the fine-tuning context it's used for preference alignment — teaching the model to prefer certain outputs over others. A simpler modern alternative is **DPO (Direct Preference Optimization)**, which takes pairs of preferred/rejected responses and directly adjusts the model toward the preferred ones, without a separate reward model. For style fine-tuning (Pass 3), neither is likely necessary. **SFT (Supervised Fine-Tuning)** — training directly on (input, desired output) examples — is probably sufficient. You provide examples of the prose style you want and train the model to reproduce that mapping.

**Cost in practical terms.** A LoRA fine-tune on 1,000–5,000 examples might run in a few hours on a rented A100. Cloud cost is roughly $10–50 for a single run, which is genuinely accessible for a small project.

**For DAVE specifically.** Pass 3 is the strongest use case. The task is stylistically well-defined (prose in a specific register, consistent with a module's tone), the output quality is directly perceptible to a human reader, and it's where a generic model most visibly shows its edges. A fine-tuned Pass 3 model for Meryton, trained on Austen prose and curated example outputs, would likely produce noticeably better sentence rhythm, period-appropriate vocabulary, and handling of social subtext than a general-purpose 7B baseline.

The training data source for Pass 3 can be entirely clean: public domain literary prose (Gutenberg) plus example outputs generated by the Sonnet construction pipeline (use Sonnet to produce high-quality examples, then LoRA-train Salamandra to approximate them). This is Feature 19b in `docs/future_features.md`, framed there around Pass 2; the same logic applies directly to Pass 3.

Pass 2 (adjudication) is a more complex fine-tuning target because the training examples are harder to produce — you need ground-truth (context_packet, action) → (outcome_json) triplets showing correct psychological reasoning. The Sonnet construction pipeline is again the natural source. This is longer-term work.

**The IP and ethics case for authors.** If a game author starts with Salamandra 7B (trained only on Common Corpus — public domain and permissively licensed material) and fine-tunes it using only material they own or have licensed, the resulting model encodes only: (a) what Salamandra learned from the Common Corpus, and (b) what the author provided. No third-party copyrighted content is latent in the pre-training.

This matters in two directions. The author can be confident the model won't reproduce someone else's copyrighted text in its outputs — a genuine concern with models trained on undisclosed internet scrapes. And the author's own content doesn't flow to parties they didn't authorize. With a commercial hosted API, content submitted in prompts may or may not be used in future training depending on the provider's data policy; this concern is common among authors and is not unreasonable (see Rachel Neumeier's public comments on this, below). A locally-deployed fine-tuned model eliminates it entirely: fine-tuning data stays on the author's (or host's) hardware, the resulting weights are the author's, and nothing leaves their control.

The practical consequence for DAVE as a platform: the combination of an ethically clean pre-trained model (Salamandra 7B / Common Corpus), author-supplied fine-tuning data, and local deployment is a meaningful proposition for an author who wants to publish a module based on their own original world. They can have confidence about provenance in both directions.

**A concrete example: the Tuyo world.** Rachel Neumeier (rachelneumeier.com) has published 13 books set in her Tuyo world, including a dedicated world companion. She self-publishes, has an active Patreon, and engages her readership directly. She has publicly expressed thoughtful skepticism about feeding unpublished manuscripts to commercial AI providers — specifically concern about whether providers actually delete submitted content — while simultaneously demonstrating openness to LLMs as tools for limited tasks. A locally-hosted DAVE instance fine-tuned on her Tuyo books under NDA would directly address the data privacy concern she has articulated. Her world has the depth (geography, cultures, magic system, political structure, extensive cast) to support rich game modules, and her fanbase is engaged enough that a Patreon post soliciting interest in interactive fiction set in the world would likely surface enthusiastic response. The approach would be: solid proof of concept first, then approach with a proposal that leads with the data sovereignty argument.

**How much text before diminishing returns on style fine-tuning.** Style is pervasive — every sentence an author writes exhibits it. You're not teaching the model facts; you're shifting its probability distribution toward her sentence rhythms, vocabulary register, and narrative distance. Research on LoRA style fine-tuning suggests the sweet spot is roughly 500–2,000 well-constructed training examples. In terms of raw prose, that's achievable from one or two novels — maybe 100,000–200,000 words. For a consistent author like Neumeier, the 13th book adds relatively little stylistic signal that the 2nd didn't already encode; early saturation is likely.

The important caveat is that raw text isn't the right format for fine-tuning. Training examples are structured as (scene context + character states → prose output) pairs, which have to be constructed from the source material — not fed sequentially. That construction step takes curation effort proportional to example count, not total word count. The world companion is particularly valuable because it provides structured world knowledge in a form closer to a DAVE context packet.

**Style vs. world knowledge saturate at different rates.** Style (voice, register, rhythm) is effectively captured from a small sample. World knowledge — geography, cultural rules, named characters and their relationships, magic system constraints — accumulates across all 13 books and the companion, and more source material meaningfully expands what the model can draw on when generating contextually grounded prose. In DAVE's architecture, however, most world knowledge properly lives in the module's seed.sql and is injected into context packets at runtime. The fine-tuned model doesn't need to *remember* that a specific character has a specific standing in a specific tribal hierarchy — it gets told. The DB-as-ground-truth principle works in the author's favor: fine-tuning can focus on style, and structured data handles facts.

**The authorial assistant use case — and where RAG is better than fine-tuning.** The questions Neumeier describes in her blog post (continuity, character presence, potentially confusing passages) are mostly retrieval problems, not style problems. "Did I forget about this character in the second half?" requires finding specific passages in the existing text. A fine-tuned model encodes probabilistic associations and will confabulate details not in training, or fail to reliably recall specific facts it was trained on. For authorial assistant tasks, RAG over the actual manuscript text would be more reliable: the system retrieves the relevant passages and the model reasons over what it was literally handed, rather than trying to recall what it learned during training.

This is a convergence of both concepts. A full system for an author like Neumeier might want two things in parallel: a LoRA fine-tuned Salamandra for style-consistent prose generation (the DAVE Pass 3 use case), and a RAG index over her complete works for the authorial assistant use case (continuity checking, world consistency, the kinds of questions she described). Fine-tuned model handles "write in my voice"; the RAG system handles "check my facts against what I've already written." These are distinct problems and the right tools differ.

The hosted service framing makes this tractable: ingest the works once (fine-tuning + indexing), author interacts with a unified interface that routes to the right tool depending on the task, the whole thing runs on hardware operated under NDA. The author never submits unpublished manuscripts to a third-party API. That's a qualitatively different privacy guarantee than any commercial hosted service can offer — not a policy, but an architecture.

**DAVE as a general author's assistant — and world companion writing.** DAVE's three-pass structure is a general pattern: parse intent, reason over structured context, render output. It isn't specific to games. For an author's assistant the passes map naturally — parse the author's question, retrieve and synthesize what the source material says, render the result as a companion entry draft or a direct answer. The game and the assistant are the same engine with different module seeds and a different Pass 3 output target.

For writing a world companion, the workflow would be: the author asks for an entry on a particular tribe, region, or character. The system retrieves all relevant passages from the indexed novels, synthesizes what's explicitly established, notes gaps or apparent inconsistencies, and drafts a structured entry. The author reviews, edits, and confirms. The confirmed entry is then added to the knowledge base — and this is the significant part. Each confirmed entry makes subsequent queries more reliable, because the companion itself becomes a source alongside the novels. As the companion is written, the system gets progressively better at answering questions about the world, because the author is canonizing details that were previously only implicit in prose.

This is the same principle as DAVE's lazy world generation: details are synthesized on demand and stored canonically on first confirmation. The world companion writing process is that pattern applied to reference material rather than game locations.

For a second volume of a world companion (Neumeier has indicated she is planning one for the Tuyo world), the system would be most useful for exactly the hard parts: cross-referencing what was said about a character or place across 13 books, identifying what's established versus what's never been pinned down, and flagging apparent contradictions between early and late books where the world evolved during writing. Those are the tasks that make companion writing laborious without assistance and that the retrieval layer handles well.

The pitch angle for this use case shifts slightly from the game use. Data sovereignty is still the hook — unpublished draft entries, internal world notes, inconsistencies an author would rather not expose to a commercial API. But the immediate value proposition is simpler and more concrete: "I will save you weeks of re-reading your own backlist." That requires no explanation of fine-tuning, quantization, or game engines. Fine-tuning for prose style is the game-module feature; the world-bible assistant is a standalone service that the same infrastructure supports. An author could get value from the second before the first is built.

**The ingestion pipeline.** Ingesting a novel series to build a structured knowledge base requires a multi-pass pipeline, because no single context window is large enough to process a full novel as one document and because extraction quality degrades over long inputs.

The first pass chunks each book into manageable segments (roughly scene breaks or ~1,500-token windows) and runs an extraction prompt on each chunk: identify all characters mentioned, their relationships to other named characters, their location, and any stated facts — output as structured JSON. This produces a stream of partial facts about many entities.

The second pass accumulates those extractions into entity records. The hard problem is **entity resolution**: "Ryo," "Ryo inGara," "the young warrior," "my tuyo," and "the boy" in different passages all refer to the same person, and the system must know that. This is genuinely difficult in fantasy fiction where naming conventions vary and point of view affects how characters are referenced. It requires either a disambiguation pass (feed the system a known character list and ask it to map references) or a fuzzy-merge step reviewed by a human.

A world companion already in existence dramatically simplifies this step. Neumeier has published a Tuyo World Companion Vol. 1, which provides a structured, author-confirmed list of characters, locations, relationships, and terminology. Fed into the system before ingestion begins, this serves as the disambiguation anchor — the pipeline can match references in the prose novels against the canonical names and entries in the companion, resolving most ambiguity without guesswork. The existing companion effectively seeds the database the same way a DAVE seed.sql does, and the extraction pipeline fills in what the companion doesn't cover explicitly.

The third pass is synthesis: consolidate accumulated facts into clean records and flag contradictions between early and late books for human review. "In book 2 Marag has three grandchildren; in book 9 she has five" is exactly what the author wants to know about, not have silently resolved.

**Could Salamandra 7B do this instead of Sonnet?** For bulk extraction (find character names in this passage, extract stated relationships), a capable 7B instruct model with good prompting and JSON enforcement is plausible — cheap, runs locally, can process the full corpus overnight. For synthesis, disambiguation, and conflict detection — tasks that require holding many facts simultaneously and reasoning about consistency — these are Sonnet-class tasks where 7B models show their limits. The natural split is the one DAVE already uses: the smaller model for mechanical extraction passes, Sonnet for synthesis and final entry generation. Sonnet handles construction; Salamandra handles the live system.

**Schema extensions needed are modest.** The DAVE schema already has character, location, faction, character_attitude, and location_connection — most of what an author's assistant requires. The additions would be: source references (which book, chapter, and passage a fact came from, supporting RAG retrieval and citation), a confidence flag on extracted facts, a conflicts table for flagged contradictions, and a character_alias table mapping variant references to canonical entity records.

**The prompt prefix architecture.** The same database, context packet assembly, and RAG retrieval layer supports different modes simply by changing the system prompt prefix:

- *Game mode:* Pass 1 parses player intent, Pass 2 adjudicates social outcomes, Pass 3 renders narrative prose.
- *Author's assistant mode:* Pass 1 parses the author's question, Pass 2 retrieves relevant DB entries and source passages, Pass 3 renders a direct answer or structured draft.
- *Companion writing mode:* Pass 2 retrieves everything known about a subject and flags gaps, Pass 3 drafts a companion entry in encyclopedic register.

DAVE's design deliberately separates what the database knows from what the LLM does with it. Swapping the prompt prefix is exactly how you repurpose that foundation without rebuilding anything.

The one addition the author's assistant mode needs that the game loop doesn't is an explicit **human confirmation step** between synthesis and writing to the database. In the game, Pass 2 output is trusted enough to commit with validation. For the world companion, every extracted or synthesized fact should go through the author before becoming canonical. That confirmation loop is where authorial judgment lives, and the system should make it easy rather than bypass it.

**Relationship taxonomies and the limits of extraction.** When extracting relationships from prose, an LLM will default to categories its training data made salient — spouse, parent, child, friend. Original fiction often has relationship categories that don't map onto those defaults. The Tuyo world distinguishes five named categories of wives with different legal standing, different rights for children, different titles, and different social expectations: Jewel Wives (primary wives), Red Ink Wives (long-term concubines with legal protections, analogous to the *placée* institution in antebellum Creole New Orleans), Summer Wives (contracts under a year), One Candle Wives (legal status for the duration of one candle), and Talon Wives (married to a group of eight soldiers, traveling with them in the field). An extraction pass without an explicit taxonomy would collapse all of these into "wife," discarding the distinctions that make the world coherent.

This is the ingestion counterpart to the same principle in DAVE's engine design: the system must be told what the world contains and how it is organized; it cannot infer it. The world companion seeds the taxonomy before ingestion begins. In the engine, relationship status is encoded as faction membership plus role — not a dedicated schema field — so the engine makes no assumptions about what relationships exist, how many parties they involve, or what legal standing they carry. A general-purpose engine cannot bake in a particular cultural model of marriage or kinship.

**Benchmarking the ingestion pipeline: known vs. unknown content.** The Wonderland module provides a useful calibration point for comparing Sonnet and Salamandra on seed generation. *Alice's Adventures in Wonderland* is Project Gutenberg text, almost certainly in Common Corpus, and therefore in Salamandra's training data. A Salamandra seed generation run on Wonderland is testing recall of pre-trained knowledge, not extraction from novel material — the model already knows the characters, relationships, and events. What the comparison measures is structured output quality, schema adherence, OCEAN inference, and goal/motivation description quality on DAVE's specific prompt format. That's valuable, but it's a different question from "can the model extract world knowledge from a text it has never seen?"

The distinction matters for calibrating expectations on the author ingestion use case. The Tuyo books are self-published with standard copyright and are not in Common Corpus — Salamandra has genuinely never encountered them. Success on Wonderland seed generation therefore sets an upper bound on Salamandra's performance, not a realistic estimate of what it will do on Tuyo. The Wonderland comparison tells you how far apart Sonnet and Salamandra are when both know the material; the gap on unknown material is likely larger. Knowing both numbers gives a realistic picture of where the smaller model is and isn't sufficient for construction work.

## Quantization Degradation and DAVE's Database Architecture

**How quantization degradation manifests.** Lower quantization makes models less precise in two ways relevant to DAVE: reasoning quality (less consistent logic, weaker causal chains in Pass 2 adjudication) and prose quality (flatter vocabulary, more formulaic sentences in Pass 3). It does not primarily manifest as factual hallucination in the traditional sense — inventing facts that aren't in the input — because DAVE supplies all facts explicitly.

**Why DAVE's architecture already addresses the hallucination problem.** DAVE was designed — from direct experience with LLM failure modes, before those failure modes had formal names — around the observation that LLMs hallucinate results and lose track of state over long exchanges. The solution was to treat the LLM not as a repository of truth but as a natural language interface to an authoritative data structure. Every fact Pass 2 needs to reason correctly — character attitudes, emotional states, locations, pending intents, item positions — is injected from the database. The LLM cannot hallucinate Darcy's attitude score toward Elizabeth because the score is in the context packet. It doesn't need to "remember" anything across turns because the database holds all state between turns.

This is the hybrid approach: use each tool for what it is genuinely good at. LLMs are good at natural language parsing, social reasoning from explicit context, and prose generation. Relational databases are good at reliable state retention, long-term consistency, and truth. The analogy to functional specialization in the brain — different areas for logic, language, and memory — is not decorative. It was the actual reasoning. E's formal background in education and linguistics includes cognitive architecture: the brain does not use a single general-purpose system, and there was principled basis for skepticism about claims that a language model could or should. A general-purpose system pressed into every role performs every role worse than a set of specialized tools.

**The asymmetry across the three passes.** The database architecture provides the strongest protection at the passes that need it most:

- *Pass 1 (intent parsing)* — narrow task, structured output, DB provides known_locations and action type vocabulary. Most robust to quantization degradation.
- *Pass 2 (adjudication)* — all facts are explicit in the context packet; the model reasons over what it is given. Quantization degradation appears as reasoning quality (are the attitude deltas psychologically coherent?) not as factual error.
- *Pass 3 (prose rendering)* — the DB constrains *what* is written but not *how well*. Prose quality is a generative capability that degrades with precision reduction. This is the pass most worth evaluating carefully when piloting Salamandra 7B against a target quantization level.

**The unforeseeable benefit.** The fact that this architecture is also robust to quantization degradation — and therefore well-suited to a small, ethically-trained local model — was not a design goal at inception. It is a structural consequence of the decision to keep the LLM stateless and the database canonical. The decision was made from practical experience, not from a theoretical position about quantization. That it turned out to be the right architecture for Phase 2 (local model deployment on consumer hardware) is a case where good engineering instincts preceded the formal justification.
