# DAVE RPG Engine — Design Document

*Brainstorm Capture · Version 0.5 · May 2026*

---

## 1. Vision and Goals

This project is an open-source game engine for text-based narrative RPGs in which diplomacy, relationships, and social dynamics are mechanically as rich as combat is in conventional RPGs. The engine uses a small, locally-running language model (LLM) as a natural-language interface and adjudication layer, while a relational database holds all canonical world state. The LLM is never the source of truth — the database is.

The design is deliberately modular so that it can eventually be adopted as an add-on narrative layer for more conventional graphical RPG engines.

Underlying design philosophy: the engine should allow players to talk, bluff, bribe, empathize, create art, build alliances, and outwit adversaries with the same expressive freedom that combat-focused games give to players who prefer action. Non-violent problem solving and conflict resolution are first-class mechanics, not afterthoughts.

### 1.1 Reference Points

- Interactive fiction tradition: Infocom games, especially later titles (Suspended, Starcross). The LLM replaces and greatly extends the NLP parser of those games.
- Tabletop RPG design: Fate system, which emphasizes narrative over random combat and allows players to resolve challenges through creative action rather than stats alone.
- Academic precedents: Façade (Mateas & Stern, 2005); the Versu social simulation system (Richard Evans / Emily Short); work by James Lester (NC State) and Mark Riedl (Georgia Tech) on interactive narrative and procedural storytelling.

### 1.2 Open Source Strategy

The engine will be released as open source. A content marketplace or paid content layer may follow, but the core engine should be freely available and community-extensible. The goal is broad adoption — including eventual integration into commercial titles — because wider adoption means more players practicing narrative, diplomatic, and creative problem-solving through games.

## 2. Core Architecture

### 2.1 Fundamental Principle

The database holds canonical world state. The LLM holds nothing permanently. All details generated by the LLM are written to the database before they become canonical facts. This ensures world consistency across sessions and players.

### 2.2 The Three-Pass Engine Loop

Each player action triggers three sequential LLM calls, each with a narrow, well-defined task and a structured JSON context packet assembled from the database. The LLM returns structured JSON in all three passes; prose is only the final output.

1. **Intent Parsing.** The player's free-text input is parsed into a structured action record: action type, target entity, item (if any), and inferred goal. Context provided: raw input, location ID, entities present, recent actions.

2. **Outcome Adjudication.** The engine assembles a full context packet from the database (game record, player record, location record, NPC record, interaction history) and asks the LLM to adjudicate the outcome as structured data: success/failure, attitude deltas, goal effects, new emotional states, narrative beat, and any new world details to store.

3. **Prose Rendering.** The engine sends the narrative beat, NPC voice parameters, setting, and game tone to the LLM and receives the final player-facing prose. This pass uses the smallest context of the three.

The engine writes all consequences from Pass 2 to the database before Pass 3 executes. The prose is always rendered from stored state, never from LLM memory.

### 2.3 Lazy World Generation

The world is seeded with skeleton data. Details are generated on demand when a player queries something not yet in the database. The engine first runs a plausibility check (is this the kind of place or person where X would exist?), then generates and stores the detail canonically. Once stored, the detail is retrieved, not regenerated.

Example: a courtyard record may not initially specify whether flowers grow there. When the player asks, the engine checks the location type and decides. If yes, that detail is written to the location record permanently — unless a subsequent event (fire, drought, significant change of state) invalidates it. Invalidation uses a state versioning or dirty-flag mechanism on affected records.

### 2.4 Player-Driven Detail Creation

**Players can call details into existence, and the engine should allow and track this.**

This is the player-facing complement to lazy world generation. In §2.3, the engine generates details on demand when the player queries the world. In §2.4, the player *asserts* a detail by acting on it — and the engine legitimises and canonicalises it.

When a player says "I fetch two glasses of negus and give one to Charlotte," they are not asking whether negus exists; they are asserting that it does and acting accordingly. Pass 2 evaluates plausibility in context (is this an assembly room where negus would be served? yes) and, if plausible, treats the assertion as a successful action. The items are created, the transaction is recorded, and from that point they are canonical — the glasses exist in the database, are tracked as held by specific characters, and have mechanical consequences (hands occupied, spill risk, social prop).

This principle applies broadly:
- **Consumable items** (a glass of negus, a dance card, a letter): created lazily on first player interaction, no prior seeding required.
- **Minor world details** (the colour of a ribbon, the name of a tune the musicians are playing, what is being served at supper): asserted by the player and confirmed by Pass 2 if plausible; written to `location_detail` or the relevant record.
- **NPC possessions and props** (a handkerchief, a fan, a pocket watch): created when first referenced in an interaction.

The plausibility check is Pass 2's responsibility. A player who asserts the existence of a loaded cannon in a Regency ballroom will find Pass 2 adjudicating failure; a player who asserts a glass of negus at the same assembly will find it created and tracked. The world is not infinitely plastic — it has internal consistency enforced by the adjudication layer — but within the bounds of plausibility, player narrative authority is respected and made canonical.

**Major items are the limit of this principle.** A sword, a horse, a significant sum of money, a plot-critical object — these should either be pre-seeded (they exist in the world and can be found or purchased) or require a narrative cost to introduce. The exact mechanism is a future design question, but the principle is: minor and consumable details appear lazily for free; significant or implausible details require grounding. This connects to future Fate-inspired outcome types (succeed at a cost, introduce a complication with a narrative price).

## 3. Data Model

Five core record types contribute to each interaction. The context assembly layer queries all relevant records and filters to action-relevant fields before building each LLM packet, keeping token usage lean.

### 3.1 Game Record (session-wide, loaded once)

Stable world parameters that color every LLM call. Fields include: genre (high fantasy, hard SF, steampunk paranormal, renaissance political drama, etc.), tone (gritty realism, fairy tale, etc.), era, cultural norms relevant to common action types, magic system rules, technology level, and narrative register. This record is referenced in Pass 3 (prose rendering) primarily, but culturally relevant norms may be included in Pass 2 when the action type warrants it.

### 3.2 Player Character Record

Tracks who the player character is as others perceive them, and who they are becoming through play. Fields include:

- Apparent status and social role
- Faction affiliations and rank within each
- Reputation scores per faction (float, −1.0 to 1.0)
- Appearance and visibly carried items (socially salient inventory)
- Personality record: Big Five trait scores (OCEAN, float per dimension)
- Ford-Nichols goal weights (weighted references to the 24-goal taxonomy)
- Maslow tier (current dominant need level, used for priority overrides)
- Emotional state (short-term modifier, separate from stable traits)
- Narrative points (running total, displayed to player)
- Skill/craft levels for creative pursuits (bard, visual artist, writer, etc.)
- Character growth event log (milestones where the personality record was updated)

### 3.3 Location Record

Fields include: location type and description skeleton, generated details store (with state version / validity flags), current occupants, social setting (public/private, formal/informal), witness count and type, and active situation flags (formal occasion, crisis, etc.). Generated details carry a validity condition — an event type that would invalidate them — so the engine can mark them stale after relevant world events.

### 3.4 NPC Record

Fields include:

- Big Five trait scores (OCEAN, stable, slow-changing)
- Ford-Nichols goal set (weighted, from the 24-goal taxonomy)
- Maslow tier (current dominant need level)
- Emotional state (short-term modifier)
- Attitude toward player (float, −1.0 to 1.0, updated by interaction)
- Attitudes toward other named NPCs (same structure)
- Faction membership and social role
- Capability beliefs (self-efficacy per relevant domain, per Ford's MST)
- Context beliefs (perceived environmental support/threat, per MST)
- Voice parameters for prose rendering: register, warmth, verbosity
- Artwork/performance history (references to art objects encountered)
- Surface motivation (visible goals and attitudes)
- Hidden motivation (actual goals, fears, and attitudes the NPC conceals; access-controlled)

### 3.5 NPC-Player Interaction History

A rolling summary (not a full log) of the player's prior interactions with each specific NPC. Regenerated and stored periodically to stay token-efficient. Captures relationship trajectory, notable events, and any private knowledge this NPC has about the player that differs from public reputation.

### 3.6 Art Object Record

Artworks are a special class of object carrying an encoded emotional payload. Fields include:

- Medium (music/vocal, music/instrumental, visual/street, visual/portrait, written/satirical, performance, etc.)
- Creator and craft level (float, affects reception quality)
- Intended emotion (primary surface emotion)
- Subtext (secondary meaning, detectable by perceptive or aligned NPCs)
- Cultural register (folk vernacular, courtly, avant-garde, etc.)
- Target audience (individual NPC, faction, public)
- Performance/display context (tavern, public square, private salon, etc.)
- Location reference if displayed in world (persists in that location's record)

## 4. Psychological Frameworks

Three established psychological frameworks provide principled structure for NPC (and player character) motivation and behavior. Their combination allows the LLM to infer behavior in novel situations without hand-coded rules for every case.

### 4.1 Big Five Personality Traits (OCEAN)

Openness, Conscientiousness, Extraversion, Agreeableness, Neuroticism. Stored as five float values per character. Stable and slow-changing. These traits determine how an NPC pursues goals and responds to social pressure — two NPCs with identical goals but different trait profiles will respond very differently to the same player action. The LLM receives OCEAN scores directly and can apply them in adjudication without further translation.

### 4.2 Ford-Nichols Taxonomy of Human Goals (MST)

Martin Ford's Motivational Systems Theory (Motivating Humans, Sage 1992; expanded in Motivating Self and Others, Cambridge 2020) provides a taxonomy of 24 universal human goals. Each NPC (and the player character) carries a weighted set drawn from this taxonomy. Goal examples include Belonging, Self-determination, Equity, Mastery, Understanding, Safety, and Social Responsibility.

MST identifies four interacting motivational components: personal goals, capability beliefs (self-efficacy), context beliefs (perceived environmental support), and emotional arousal. All four are represented in the NPC record. The concept of "equipoise" (motivational equilibrium) is a useful design hook: an NPC in equilibrium behaves very differently from one under stress, providing principled behavioral variation without manual coding.

The Ford-Nichols goal taxonomy distinguishes between within-person goals (aimed at changes in oneself) and person-environment goals (aimed at changes in the external world), and applies an approach-avoidance dimension across both. This provides a structured basis for predicting how an NPC will respond to threats or opportunities.

Note: A physical copy of Ford's textbook is available for reference. Scans of relevant taxonomy tables can be incorporated into engine documentation.

### 4.3 Maslow's Hierarchy of Needs

Used as a priority-override mechanism rather than as a strict psychological theory. When lower-tier needs (physiological, safety) are threatened, higher-tier goal pursuit is temporarily suppressed. This provides a principled rule for NPC behavior under duress without requiring manual encoding of every scenario. Implemented as a `maslow_tier` field in each character record, updated dynamically by world events.

## 5. Social and Narrative Mechanics

### 5.1 Creative Action Resolution

Players can attempt any action and receive a reasoned response grounded in who the NPC actually is. There are no dialog trees with three options. The adjudication pass evaluates creative actions — offering a gift, making an argument, performing art, revealing information, making a threat — by checking NPC traits, goals, emotional state, current Maslow tier, and contextual factors (witnesses, setting, prior history) before generating an outcome.

Outcomes are not binary. A poorly-executed gift is received differently than a well-timed one. A satire aimed at the wrong audience may confuse rather than persuade. A sentimental ballad performed to a grief-stricken NPC may have unpredictable effects depending on their emotional state.

### 5.2 Art as Social Action

Art is defined for engine purposes as a creation that evokes an emotion. Art objects carry an intended emotional payload (and optionally a subtext) that interacts with each NPC's trait profile, goals, and cultural register to produce individuated reception responses. Key design principles:

- Craft level modifies reception quality, not outcome type. A poorly executed ballad still evokes something — perhaps embarrassed sympathy or amusement rather than the intended nostalgia. Failure states are narratively rich rather than binary.
- Subtext is detectable selectively. High-Openness or politically-aligned NPCs may detect the critique embedded in a folk song while others simply enjoy the melody. This rewards player investment in artistic skill and subtlety.
- Public art (street art, tavern performance) is resolved as a broadcast interaction: one artwork adjudicated against a sample of NPCs from the target audience, producing aggregate faction mood shifts.
- Art objects persist in the world. A mural remains on a wall. A ballad performed at a festival may be remembered by NPCs who were present. Art references appear in NPC memory records and location records.
- Player artistic careers (bard, street artist, propagandist, poet) are supported through the skill/craft level fields in the player character record, without requiring rigid character classes.

### 5.3 The Narrative Point System

The player character has a personality record (OCEAN + Ford-Nichols goals) that enables the engine to model what a character with those traits and goals would naturally do in any situation. Narrative points accumulate when the player acts consistently with their established character, and are spent when they act against type.

The system is designed not to punish creativity. Acting against character type costs narrative points but is always permitted. At dramatically appropriate moments, acting against type in a principled way may trigger a character growth event rather than merely costing points (see 5.4).

NPC art, music, and emotional influence affect the player character's temporary emotional state, which shifts the "what would this character naturally do" baseline for the duration of the effect. An inspiring battle hymn makes aggressive action more "in character" momentarily; a melancholy ballad may make a grieving character's impulsive action more understandable and cheaper in narrative terms.

The running narrative point total is displayed to the player. The engine surfaces the character's internal state as prose suggestion rather than a mechanical prediction: "you feel drawn to caution" rather than "the model recommends diplomacy." Players can read and act on these suggestions, or ignore them.

### 5.4 Character Growth Events

When a player acts against established character type in a dramatically significant and contextually appropriate way, the engine can recognize this as a character growth event rather than a narrative point expenditure. At these milestones, the personality record is updated to reflect who the character is becoming — not who they were seeded as. This makes genuine character development a mechanical reality rather than a cosmetic choice.

NPC personality records also drift slowly in response to significant events. An NPC with repeated positive interactions with the player may gradually shift trait scores or goal weights, making the world feel genuinely alive rather than statically scripted.

## 6. Player Agency and Accessibility

### 6.1 LLM Upgrade of Player Expression

Just as an FPS player does not need real firearms skill and a fantasy player does not need real magic ability, a player whose character is a gifted diplomat should not be disadvantaged by their own limited eloquence. When a player's character has high persuasion capability or relevant social skills, the prose rendering pass can be instructed to render the character's speech at the level the character would actually deliver it, even if the player's raw input is plain or ineloquent.

The player's intent drives the action; the LLM upgrades the surface expression to match the character's established ability. The upgraded version is shown to the player so they experience what it is like to be that character at their best.

Design variants for this feature: silent upgrade (always on), dual display (player text and character text shown together), or optional on player request. The reverse also applies: players who are personally more eloquent than their character can have input toned down to character level, which is its own creative constraint.

Accessibility note: for players with communication disabilities or who are writing in a non-native language, the upgrade option is a genuine accessibility feature and should be documented and designed as such from the start, not retrofitted later.

### 6.2 Character Classes and Skill Growth

The engine intentionally avoids rigid character classes. Growth is possible across many dimensions — social, artistic, diplomatic, intellectual, physical — for both player characters and NPCs. Skill and craft levels in the player character record support specialist paths (bard, artist, diplomat, scholar) without locking players into them at character creation. Character growth events provide the primary mechanism for meaningful development.

## 7. Local LLM Selection

The engine targets a small, locally-running LLM for privacy, offline capability, and cost. All data remains on-device. Structured JSON output reliability is the primary selection criterion, since the three-pass architecture depends on parseable structured responses.

Current primary target: Salamandra 7B via Ollama (Barcelona Supercomputing Center). Trained exclusively on Common Corpus (public domain and openly licensed sources), which eliminates ethical concerns about copyrighted training data and makes the engine suitable for authors developing modules from their own unpublished work. Salamandra 7B is Apache 2.0 licensed, available via Ollama, and supports multilingual inference — an asset for non-English modules. Structured JSON output reliability is to be validated against DAVE's Pass 2 requirements during Phase 2 testing; fine-tuning on DAVE-format adjudication examples is a planned optimisation path if baseline quality is insufficient.

Fallback: Mistral 7B via Ollama. Approximately 90% first-try JSON accuracy with minimal retry overhead, 6–7 GB RAM, Apache 2.0 licensed. 32K context window is sufficient for current context packet sizes.

Phi-4-mini (3.8B) offers the smallest footprint for resource-constrained hardware but has shown less reliable structured output compliance in the Phi model family and is not recommended as the primary model for this architecture.

Context packet token budget: with three LLM passes per player action and typical context packets in the 500–2000 token range per pass, a player action costs approximately 1500–6000 tokens. This is well within the capacity of the recommended models. Token optimization strategies include: single-letter OCEAN keys, goal IDs as short references to a lookup table, controlled vocabulary enums for emotional states and narrative beats defined at session start, and action-relevant filtering of the game record before packet assembly.

## 9. Skills, Learning, and Knowledge Objects

### 9.1 Skill Representation: Continuous Floats and Social Status

Skill levels are stored in the database as continuous float values representing genuine capability. This allows fine-grained progression without arbitrary step discontinuities and supports nuanced adjudication — a skill float of 0.71 and one of 0.73 can produce subtly different outputs, where a step system would treat them identically.

Social recognition of skill — journeyman, master, adept, and equivalent designations — is stored separately as faction status, not derived automatically from the skill float. These designations are conferred, withheld, and revoked by factions (guilds, academies, religious orders, professional communities) according to their own criteria, politics, and cultural norms. The gap between real capability and recognized status is a design feature, not a problem to be resolved. A self-taught artist whose float exceeds a guild journeyman's may lack access to certain commissions, spaces, or social standing without recognition. An incompetent practitioner may hold a master designation through patronage. These tensions are sources of meaningful narrative conflict that emerge naturally from the data model.

Named thresholds (novice, apprentice, journeyman, master, virtuoso, or domain-specific equivalents) are defined per skill domain as landmark values on the underlying float, used by the prose rendering pass to calibrate how the world describes and responds to the character's level. The underlying float drives adjudication; the threshold label drives narrative texture.

### 9.2 Learning Mechanics

Three pathways modify skill floats, each with different rate and ceiling characteristics:

1. **Formal instruction by a character.** The teacher's skill float in the domain sets the transmission ceiling — you cannot learn beyond what the teacher knows. A separate teaching capability attribute on the NPC record (distinct from domain skill, since expert practitioners are not always effective teachers) scales how much of that ceiling is reached per session. The learner's trait profile modulates uptake: high Openness accelerates initial acquisition; high Conscientiousness improves retention and reduces decay.

2. **Reference objects (texts, diagrams, notation).** Knowledge objects carry a skill payload analogous to the emotional payload of art objects. Gain rate is slower than formal instruction and subject to a prerequisite floor — a treatise on advanced counterpoint does nothing for someone who cannot yet read music. The reference record specifies the author's skill level at time of creation, the quality and completeness of instruction, the required prerequisite skill level for effective use, and the domain and medium.

3. **Independent practice.** Produces the slowest gains with the lowest ceiling. Without external feedback, self-correction is limited. Practice accumulates in the character record as a practice log (domain, session count, recency) that the engine uses to apply gradual float increments. Conscientiousness is a significant moderator here — high-Conscientiousness characters practice more effectively and consistently.

All three pathways can combine: a character who studies a reference, receives occasional instruction, and practices regularly between sessions will advance faster than any single pathway alone. Skill floats also decay slowly without practice — a domain that has not been exercised recently loses ground, at a rate modulated by how deeply established the skill is.

### 9.3 Reference Object Record

Reference objects (books, scrolls, annotated diagrams, musical scores, technical manuals, recipe collections) are a distinct record type parallel to art objects. Fields include:

- Domain: the skill domain the reference addresses
- Author skill float at time of creation: sets the transmission ceiling
- Instruction quality: completeness and clarity (float, affects gain rate)
- Prerequisite skill level: minimum float required to use effectively
- Medium: written text, annotated diagrams, musical notation, oral tradition transcript, etc.
- Cultural register: affects which characters can access or value the reference
- Location reference: where the object currently exists in the world
- Condition: physical state, affecting usability (pristine, worn, damaged, fragmentary)

Like art objects, reference objects persist in the world. A library is a location record with high-density reference object associations. A character who authors a reference object creates a persistent world artifact that others can learn from — this is a meaningful form of legacy and influence, especially for NPC scholars, masters, and faction knowledge-keepers.

### 9.4 Tools, Materials, and the Medium-as-Message Principle

Tools and materials affect creative output through two distinct mechanisms that must not be conflated.

Technical quality is the simpler of the two. A fine instrument, a sharp chisel, quality pigments — these raise the execution ceiling for a given skill float. They do not raise the underlying skill; they expand what that skill can produce in the moment. Tool and material quality are float fields on item records, included in the adjudication packet when a creative action is being resolved. Poor tools constrain even high skill; excellent tools allow high skill to express itself fully.

Expressive register is the more consequential mechanism. The choice of medium and materials is itself a communicative act, independent of technical quality. Spray paint on rough concrete carries cultural meaning that oil on gallery linen cannot — accessibility, urgency, impermanence, the assertion of presence in public space. A handwritten letter on worn paper speaks intimacy and effort differently than a printed document of identical content. Simple materials used deliberately by a skilled practitioner are not a handicap; they are part of the message.

In the art object record, material expressive register is a field separate from tool technical quality. During adjudication, the LLM evaluates both: technical quality as a modifier on execution ceiling, and expressive register as an additional input to NPC reception — interacting with the NPC's cultural background, Openness score, and the artwork's intended audience. A high-Openness NPC may find the deliberate roughness of street materials more compelling than polished gallery work. A conservative courtier reads the same choice as crude. Neither response is wrong; both are principled given the NPC's record.

Material literacy — the ability to choose materials whose register serves the work's intent and audience — is itself a learnable skill, stored as a separate float in the character's skill record. A technically accomplished artist with low material literacy produces well-executed work that speaks in the wrong register for its context. High material literacy without technical skill produces conceptually coherent but poorly executed work. Both dimensions matter, and both are subject to growth through the learning mechanics described above.

### 9.5 Artwork Quality as a Multi-Dimensional Record

A finished artwork is not evaluated on a single quality float. The art object record carries three distinct quality-related fields:

- Technical execution level: how well the craft float was expressed given tool quality and conditions
- Material expressive register: what the chosen medium says independently of execution
- Medium-message coherence: how well the material register serves the work's intended emotion and subtext — a field the LLM is well-positioned to evaluate during adjudication

NPC reception adjudication draws on all three, weighted by the NPC's trait profile and cultural background. A work with low technical execution but high coherence between rough medium and raw emotional content may move an NPC more than a technically perfect work in an ill-chosen register.

## 10. Faction Records

Faction records provide an efficient structure for properties shared across groups of characters, avoiding redundant storage on individual NPC records and providing a clean target for group-level effects (broadcast art, political speech, reputation shifts). They are also the home of socially-mediated skill recognition (see Section 9.1).

### 10.1 Faction Record Fields

- Faction name, type (guild, religious order, political faction, noble house, criminal organization, scholarly academy, etc.), and brief description
- Core Ford-Nichols goal set: factions have collective motivational profiles just as individuals do. Ford's Group Motivational Systems Theory (an extension of MST) directly addresses collective motivational patterns and is applicable here.
- Dominant cultural register and norms: what this faction considers appropriate behavior, acceptable art, legitimate skill, and so on
- Maslow-equivalent threat level: a faction under existential threat behaves differently than one in comfortable power — this field drives collective behavioral shifts analogous to individual Maslow tier overrides
- Reputation toward the player (float, −1.0 to 1.0)
- Reputation toward other named factions (same structure, stored as a relation table)
- Membership roster with role and tier: references to NPC records, with faction-conferred status designations stored here rather than on the NPC record
- Internal hierarchy: power structure, decision-making roles, succession rules
- Faction-specific knowledge and secrets: information held collectively that may be shared with members or revealed through play
- Skill recognition thresholds: the float values at which this faction confers each status designation, and any additional criteria (sponsorship, examination, political approval)

### 10.2 Faction Records and Broadcast Interactions

Faction records provide the clean resolution target for broadcast social actions. When a player performs public art, gives a political speech, or circulates a pamphlet, the adjudication pass samples NPCs from the target faction's membership roster, resolves individual reception for each sampled NPC, writes attitude deltas to those NPC records, and computes an aggregate delta for the faction's reputation-toward-player field. Both levels update simultaneously: individual courtiers shift in their personal attitudes, and the court's collective mood shifts as a faction. This is sociologically realistic — a satirical street mural changes specific people and ambient public sentiment at the same time.

### 10.3 Faction Tensions as Narrative Driver

The gap between a character's skill float and their faction-recognized status is a natural source of narrative conflict and motivation. A character who has outgrown their recognition has a concrete Ford-Nichols goal (equity, mastery, social recognition) being frustrated. A faction that withholds recognition for political rather than merit-based reasons has a detectable goal profile that a perceptive player can identify and potentially exploit, work around, or challenge. These tensions emerge from the data model without scripting.

## 11. Open Skill Taxonomy

The engine does not maintain a canonical list of skill types. Skill specificity is encoded in the natural language term used to name the skill. "Painting" is broad and applies across many contexts at moderate boost. "Abstract pointillist painting" is narrow and applies with full force in its specific context, degrading gracefully as the situation moves away from it. The string stored in the character record is the skill definition.

The LLM reads the semantic breadth of the skill term during adjudication and evaluates its relevance to the current context without a lookup table. This is only tractable with LLM adjudication — previous attempts at implementing flexible skill systems required restricted taxonomies to make matching computable. The semantic relationship between a narrow skill term and a specific situation is something the LLM evaluates naturally from the meaning of the terms.

The same principle extends to object properties. Rather than storing boolean taxonomy flags on every object record (is_container, is_conveyance, is_food, is_weapon, etc.), objects carry natural language descriptions and the LLM infers relevant properties from context at adjudication time. Whether a ceramic jug is a container, an improvised weapon, a trade good, or a percussion instrument depends on what the player is attempting, and the LLM evaluates that without exhaustive pre-enumeration. General semantic principles replace hand-coded property lists throughout.

This represents one of the fundamental architectural shifts that LLM adjudication makes possible. The restricted taxonomy approach was not a failure of design in earlier implementations — it was the only tractable path without runtime semantic inference. The engine can now store descriptions and defer classification to the adjudication pass.

## 12. Development Phases and Tooling

### 12.1 Phase 1: Prototyping in Claude Sonnet

Initial development and iteration will use Claude Sonnet rather than a local small model. Sonnet has sufficient reasoning capacity to handle underspecified or ambiguous situations gracefully, which means early failures will manifest as logic and data structure failures rather than model capability failures — the right kind of failure to see first. Once query templates and data structures are stable, porting to a local model will expose what needs tightening for a less capable inference engine. The gap between Sonnet and Mistral 7B is a known, plannable quantity; unknown gaps in the data model are not.

Sonnet testing also provides an opportunity to generate synthetic ground-truth examples of correct adjudication that can later be used to evaluate whether the local model performs comparably. These should be logged systematically during the prototyping phase.

### 12.2 The Meta-Channel Convention

During testing and development, a meta-channel convention allows design and debugging communications to coexist with gameplay in the same session. Non-game content is enclosed in angle brackets: `<show me the record of the current location>`. The engine treats angle-bracket content as a design or debug instruction, handled at the outermost layer before intent parsing runs.

Useful meta-channel commands during iteration:

- `<show record: [entity]>` — display the current database record for a named entity
- `<explain last adjudication>` — expose the LLM's reasoning for the most recent outcome
- `<show context packet>` — display the JSON packet sent to the LLM for the last pass
- `<override: [field] = [value]>` — directly set a field value for testing specific scenarios
- `<save state>` — checkpoint the current database

The explain command is particularly valuable: asking the engine to articulate its adjudication reasoning exposes whether context packets are being assembled and used as intended.

### 12.3 SQLite as Interchange Format

SQLite is the designated interchange format between the Sonnet prototyping phase and the local LLM implementation phase. It is local, zero-infrastructure, well-supported across all candidate implementation languages, human-inspectable, and portable. The record types in this document map naturally to relational tables. A schema version number should be embedded from the first implementation; migration scripts should be maintained alongside the schema in the project repository from day one.

The interactive seeding process should include a lightweight consistency check after each addition: before writing a new record, the engine verifies it against existing records for contradictions. This is especially important during Netherfield seeding, where a large number of socially interconnected characters and locations must remain mutually coherent.

### 12.4 Hidden Motivation as a Schema Field

A distinction between visible and hidden NPC attributes is necessary. Surface motivation — the goals and attitudes an NPC presents to the world — is what the player can observe and what low-insight interactions draw on. Hidden motivation — the actual goals, fears, and attitudes the NPC conceals — is what the adjudication pass uses for truthful outcome computation and what high-insight or investigative player actions may gradually reveal.

This distinction is essential for the locked room mystery module, where every character has a surface account and a hidden one. It is also present in Pride and Prejudice: Wickham's surface Agreeableness and his actual character are different records. Darcy's actual regard for Elizabeth is hidden motivation for much of the novel. Hidden motivation should be added to the NPC record schema as a parallel structure to the visible goal and attitude fields, with an access control flag governing when and how it becomes available to the player.

## 13. Proposed Test Modules

Three test modules are proposed, each approaching the engine from a distinct angle and using a distinct seeding method. Implementation begins with the cat scenario as the most bounded first pass, followed by the Netherfield Ball, then the locked room mystery. Together they constitute a strong proof of concept for engine generality.

### 13.1 Seeding Patterns

- **Interactive generative seeding** (I Am a Cat). Conversation-driven world-building with Sonnet producing structured records directly. Best for original or small-scope worlds. The world is small enough to be exhaustively seeded before the first play session.
- **Reference-assisted seeding** (Pride and Prejudice). Draws on external sources: public domain primary text for voice and detail, OER historical reference for period-accurate material culture and social context.
- **Structured notes import** (locked room mystery). Pre-existing structured character documentation maps almost directly onto the NPC record schema. The consistency enforcement mechanism gets its most rigorous workout here — the player is actively trying to find contradictions.

### 13.2 I Am a Cat (Module 1 — First Implementation)

Source material: original. Player character: a bored domestic cat in a townhouse, middle of the night. Goals: find toys, check the food dish, attempt to wake the humans. A comedic module that stress-tests object interaction, speech filtering, and the engine's handling of radically asymmetric communication.

Speech rendering operates under permanent constraint in both directions. Player input is parsed normally by the intent layer — the engine understands exactly what the cat is trying to accomplish — but all player speech output renders as meow variants calibrated to emotional state and urgency. Human speech is filtered through feline comprehension: rendered mostly as acoustic impression and tonal contour, with a small vocabulary of semantically salient words breaking through. "Treats" penetrates clearly. "No" penetrates partially and inconsistently. "It's three in the morning" does not penetrate at all.

Object interaction mechanics get thorough and comedic exercise. The LLM evaluates whether objects qualify as toys by feline criteria rather than human taxonomy. Human NPC motivation covers a specific adjudication problem: sleepy, annoyed humans constitutionally unable to fully ignore a sufficiently persistent cat despite strong competing rest goals.

This module requires none of the social status, faction, or reputation mechanics of the other modules. It exercises object interaction, environmental exploration, lazy world generation, emotional state modeling, and the speech rendering filter. The contrast with the social modules demonstrates that the engine is a general narrative engine, not a social simulation with other features bolted on.

The boredom score introduces a time-pressure mechanic without combat: the cat's goal is to remain engaged until morning, and boredom is the failure condition. This is a reusable design pattern — internal state degradation as narrative stakes, replacing hit points.

### 13.3 Pride and Prejudice: The Netherfield Ball (Module 2)

Source material: Jane Austen, Pride and Prejudice (1813, public domain). Player character: Elizabeth Bennet. Goal: navigate the first ball at Netherfield, help Jane make a good impression, and manage the social damage caused by the rest of the Bennet family and the initial friction with Mr. Darcy.

This module exercises almost every social mechanic simultaneously: status, reputation, faction dynamics, art as social performance (dancing, piano), material register (dress, arrival, comportment), the gap between recognized status and actual worth, proxy-goal mechanics (helping Jane), and the narrative point system. There is no combat.

NPC profiles of note:

- Mrs. Bennet: Ford-Nichols goal profile dominated by resource acquisition and social integration. High Neuroticism. Demonstrates how misaligned individual goal profiles within a faction produce collective reputational damage.
- Mr. Darcy: Low Openness at game start. High Conscientiousness, high Neuroticism around social propriety. His arc is the story of Openness developing under Elizabeth's influence.
- Jane Bennet: The NPC the player actively helps, introducing a proxy-goal mechanic.
- Wickham: High surface Agreeableness masking low underlying trustworthiness. A primary exercise of the hidden motivation schema field.

### 13.4 Locked Room Mystery in a Science Fiction Setting (Module 3)

Source material: original, partially written, with extensive pre-existing structured character notes. The designer holds detailed documentation of surface motivation, hidden motivation, and character histories for all major characters. This maps directly onto the NPC record schema and represents the structured notes import seeding pattern.

The science fiction setting offers a different set of genre conventions for the game record, demonstrating engine adaptability across genre. The locked room format is a rigorous test of consistency enforcement — all alibis, secrets, and factual claims must remain coherent across the full investigation as the player probes for contradictions. The hidden motivation schema field is load-bearing throughout.

## 14. Public Domain Strategy and Educational Use

The choice of Pride and Prejudice as a primary test module is partly genre-motivated — Regency social mechanics are an ideal exercise for this engine — and partly strategic. Austen's works are fully in the public domain, enabling unrestricted use of primary text for seeding, quotation, and voice calibration.

The adaptation tradition around Austen also provides a ready demonstration of the engine's genre-flexibility argument. The engine can perform genre transpositions parametrically — by modifying the game record's genre, tone, and setting fields — without re-authoring the social logic. This is a compelling demonstration for potential contributors and adopters.

The educational use case is genuine and worth developing explicitly. An engine that allows students to play as Elizabeth Bennet navigating Regency social structures, investigate a mystery requiring understanding of character motivation, or experience historical social dynamics through interactive fiction has real pedagogical value. Literature and history courses are natural early adopters.

All period reference material used in seeding should be documented with source and license. Preferred sources are Creative Commons licensed or in the public domain: OpenStax humanities texts, Internet Archive open collections, university press open-access imprints, and Wikimedia Commons for visual reference. This documentation standard should be established in the project contributor guidelines from the start.

## 15. Additional Module Concepts

### 15.1 Sherlock Holmes and Victorian London

The Sherlock Holmes canon is fully in the public domain, with the same rich adaptation tradition as Austen. The additional asset here is an unusually rich corpus of digitized Victorian primary sources: census records, court proceedings, newspaper archives, Charles Booth's poverty maps, medical records, trade directories, and parish records. These materials allow world seeding at a granularity that makes a location feel genuinely inhabited.

Holmes's methods are explicitly probabilistic reasoning presented in deductive language. A module built around Holmesian investigation is teaching Bayesian inference and evidence-based reasoning whether it labels itself that way or not.

### 15.2 Ada, Charles, and Florence: A Statistics Learning Module

A later-phase module explicitly designed for undergraduate statistics instruction, built around Ada Lovelace, Charles Babbage, and Florence Nightingale as player-adjacent characters in a mildly ahistorical but entertaining Victorian setting. Players gather data, reason from evidence, and solve problems using statistical tools, motivated by narrative stakes rather than grades.

Florence Nightingale is a historically underused public figure given that her coxcomb diagrams are among the most consequential data visualizations in history and her work on hospital mortality statistics directly shaped public health policy. This module extends the educational use case argument considerably: it is explicitly designed for a curriculum context, built entirely on public domain figures and materials, and addresses a documented pedagogical gap.

### 15.3 Language Learning: You've Just Moved to Paris

The engine's speech rendering pass supports a language learning mode in which the player's halting attempts in the target language are rendered at a plausible proficiency level, and NPC responses are calibrated to the player's current skill float.

Scenario: the player has just moved to Paris. First language is English; target language is French. Goal: find and rent an apartment before living in a hotel drains the bank account. Bank account balance is a world-state float with a daily decay rate representing hotel costs. The player is motivated to improve their French not by experience points but by material consequence — a more honest model of why language learning matters than gamified reward systems.

## 16. Narrative Injection and Internal State Rendering

### 16.1 The Narrative Injection Mechanism

The engine supports a designer-controlled information revelation mechanism: stored canonical facts are surfaced to the player at designer-specified trigger conditions, wrapped in a narrative framing device. The mechanism is the same regardless of framing: a fact stored in the database is tagged with a trigger condition (time elapsed, location entered, NPC encountered, player action taken, or designer-set flag) and a revelation record specifying how it surfaces. When the trigger fires, the engine retrieves the fact and passes it to the prose rendering pass with the configured framing template. The result is information that feels discovered rather than dispensed.

### 16.2 Narrative Voice Configuration

The grammatical person used in narrative injections and internal state comments is a configurable field on the revelation record and on the game record's prose rendering settings. Options include:

- **First person:** "I remembered", "I felt a sudden unease"
- **Second person:** "You remember", "You feel drawn toward caution" — the standard interactive fiction convention
- **Third person:** "She remembers", "He felt the familiar weight of the choice"
- **Distanced or atmospheric:** "A fragment of memory surfaced", "Something about the room felt wrong"

The choice of person affects the player's relationship to the character and should be a deliberate game design decision, not a default.

### 16.3 Internal State as Prose Rather Than UI

Internal state floats — boredom, hunger, anxiety, affection, suspicion, fatigue, language confidence — can be surfaced to the player either as numeric UI indicators or as in-world prose commentary generated by the rendering pass. The game record holds a configuration flag per state float specifying which rendering mode to use.

The cat scenario illustrates why prose rendering is often preferable to numeric display. A boredom float of 0.82 communicates nothing immersive; "Oh, this is SOOOO boring" communicates the same information in a way that is character-appropriate, tonally consistent, and narratively integrated. For analytically-framed modules or developer/testing contexts, numeric display remains available.

## 17. Open Questions and Next Steps

- Database schema: full table definitions for all record types including hidden motivation fields, revelation records with trigger conditions, internal state float configuration, reference objects, faction records, and faction-NPC membership relations.
- Context assembly layer: field selection per pass including when hidden motivation is surfaced to adjudication, and how revelation trigger conditions are evaluated each turn.
- State versioning / dirty-flag mechanism for invalidating generated details after world events.
- Skill decay rate: formula and moderating variables for gradual float reduction without practice.
- Practice log structure: tracking and applying incremental gains from independent practice.
- Narrative point calculation: formula for consistency scoring and growth event detection.
- Art and broadcast reception adjudication: NPC sampling strategy and faction mood shift aggregation.
- Session-level LLM priming: loading controlled vocabularies once per session to reduce per-call token overhead.
- Character growth event detection: criteria and triggers for personality record updates.
- LLM upgrade and language scaffolding feature: implementation for both eloquence upgrade and language learning proficiency rendering.
- Ford-Nichols taxonomy: incorporate full 24-goal list from physical textbook. Scans available.
- Material literacy skill: domain definition, learning pathway, adjudication integration.
- Condition field for reference objects: degradation mechanics and triggering events.
